# ADR-001: Migrate from Gatsby to Astro

## Status

Accepted

## Date

2026-05-16

## Context

The current site is built on Gatsby 2.x (released ~2020), forked from overreacted.io. The Gatsby ecosystem entered maintenance mode after Netlify's 2023 acquisition and subsequent team layoff. The plugin ecosystem is stagnant, build times are slow (~60–120s), and the stack ships unnecessary JavaScript for a fully static markdown blog. The site has 9 posts in two languages (EN + PT-BR), uses plain `.md` files (not MDX), and has no backend or authentication requirements. The owner wants a 1:1 migration with no new features in V1.

## Decision

Migrate to **Astro** using the official Gatsby → Astro migration guide. All 18 markdown files are preserved as-is. Features are replicated 1:1: bilingual routing (EN + PT-BR), syntax highlighting (Prism), image handling, RSS feed, social sharing, AdSense, dark mode, and newsletter confirmation pages. Fork artifacts from overreacted.io (RSS copy text, manifest branding, `now.json` alias, `discussUrl`) are treated as bugs and corrected during migration.

## Alternatives Considered

### Alternative 1: Next.js App Router

- **Description**: Migrate to Next.js 15 with App Router, using `gray-matter` + `next-mdx-remote` for markdown processing
- **Pros**: React familiarity (existing components port with minimal changes); Vercel-native deployment; strong ecosystem; no new component syntax to learn
- **Cons**: Ships 80–150 KB of JS per page by default for a fully static site; requires manual content schema setup; no built-in content collections; more boilerplate for static markdown rendering
- **Why rejected**: Overcomplicated for a content-only blog; adds JavaScript overhead with no functional benefit; Astro's remark plugin ecosystem overlaps significantly, reducing the actual migration delta

### Alternative 2: Eleventy

- **Description**: Ultra-minimal static site generator with near-zero JavaScript
- **Pros**: Fastest builds, zero JS, simple templates
- **Cons**: No React ecosystem; complete rewrite of all components; no Islands model for future interactivity; steep template language learning curve
- **Why rejected**: Higher migration cost than Astro with no practical advantage for this use case

## Consequences

### Positive

- Build time drops from ~60–120s to ~8s
- Zero JavaScript shipped by default (vs. ~300 KB with Gatsby 2)
- Lighthouse Performance score expected to reach ≥ 95 (vs. ~70–80 current)
- Content collections provide type-safe frontmatter validation via Zod
- Shared remark/rehype plugin ecosystem minimizes rewriting syntax highlighting, autolinks, smartypants
- Fork artifacts corrected, site fully rebranded to henriquetavares.com

### Negative

- `.astro` component syntax is a new mental model (not JSX)
- Interactive React components (Share, Toggle) require explicit `client:` hydration directives
- i18n routing must be hand-rolled (no `gatsby-plugin-i18n` equivalent in Astro)

### Risks

- **Migration stall risk**: If `.astro` syntax proves too unfamiliar, fallback to Next.js App Router is available with similar content capabilities. Mitigation: set a 2-week checkpoint; if blocked, switch target framework.
- **i18n complexity**: Bilingual filesystem routing is the most complex piece. Mitigation: implement and validate routing before migrating post content.
- **Relative image assets**: Co-located images (`./image.gif`) must survive the asset pipeline change. Mitigation: verify image rendering for all 9 posts before launch.

## Implementation Notes

1. Start with `npm create astro@latest` using the blog starter
2. Define a content collection schema in `src/content/config.ts` with Zod for `title`, `date`, `spoiler`, `tags`, `updateDate`
3. Move `src/pages/<slug>/index.md` → `src/content/posts/<slug>/index.md` (and `index.pt-br.md`)
4. Implement language routing: `/` and `/en/` for index, `/<slug>/` (PT-BR default) and `/en/<slug>/` for posts
5. Port React components as Astro components where possible; use `client:visible` for `Share` and `Toggle`
6. Validate RSS feed output against `henriquetavares.com` (not overreacted.io)
7. Run Lighthouse CI before launch; target ≥ 95
