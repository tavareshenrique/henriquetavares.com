# ADR-003: Use Astro Content Collections with Co-Located Post Assets

## Status

Accepted

## Date

2026-05-16

## Context

The current Gatsby blog stores posts under `src/pages/<slug>/`, with each post directory containing `index.md`, `index.pt-br.md`, and any relative images or other assets used by those Markdown files. The PRD requires preserving all 18 Markdown files, preserving existing public routes, and keeping Markdown as the source of truth. Astro supports content collections with frontmatter validation, but the migration must avoid breaking relative asset references or forcing large content rewrites.

## Decision

Move posts to `src/content/posts/<slug>/index.md` and `src/content/posts/<slug>/index.pt-br.md`, preserving co-located post assets inside each slug directory. Define a single content collection schema for required frontmatter and derive language, slug, route, and translation pairing from filename plus directory name.

## Alternatives Considered

### Alternative 1: Keep Posts Under `src/pages/<slug>/`

- **Description**: Preserve the current Gatsby directory tree and configure Astro around it.
- **Pros**: Lowest file movement; fewer path changes during migration.
- **Cons**: Bypasses Astro content collection conventions; weakens frontmatter validation; mixes source content with route files.
- **Why rejected**: The migration should adopt Astro's content model while preserving authoring ergonomics and co-located assets.

### Alternative 2: Split Content by Locale

- **Description**: Store posts as `src/content/posts/en/<slug>.md` and `src/content/posts/pt-br/<slug>.md`.
- **Pros**: Locale is explicit in the path; easy to list by language.
- **Cons**: Splits translation pairs across directories; breaks the current mental model of one folder per post; complicates co-located assets shared by translations.
- **Why rejected**: The existing paired-file model is simpler for a small bilingual blog and better preserves post-local assets.

## Consequences

### Positive

- Preserves the current authoring model: one directory per post, paired language files, local assets nearby.
- Enables Astro/Zod validation for frontmatter.
- Keeps language and route derivation deterministic.
- Reduces migration churn for Markdown image references.

### Negative

- Requires a small content helper layer to derive route metadata from file paths.
- Requires auditing Markdown links that currently point to `/pt-br/...`.

### Risks

- **Relative asset risk**: Markdown image references could break after moving content. Mitigation: include image-heavy posts in the migration checklist and content audit.
- **Route derivation risk**: Incorrect filename parsing could break language routing. Mitigation: content audit must assert 18 entries and 9 paired slugs.

## Implementation Notes

The TechSpec should define a small `PostEntry`/`PostRoute` domain model and centralize route generation in one helper module.
