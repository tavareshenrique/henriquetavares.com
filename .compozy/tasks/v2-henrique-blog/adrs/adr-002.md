# ADR-002: Use Strict Parity Maintenance Reset for V1

## Status

Accepted

## Date

2026-05-16

## Context

The V1 migration must move the blog away from Gatsby without turning the project into a redesign, content relaunch, or reader-feature expansion. The approved idea already removes unused surfaces (RSS, AdSense, newsletter confirmation pages) and focuses the work on preserving the current Markdown blog experience. During PRD clarification, the selected product outcome was a maintenance reset with very strict URL and visual parity, launch validation through parity checks, URL audit, and Lighthouse thresholds, plus one targeted UX correction: dark/light mode should respect the user's system preference while allowing manual changes.

## Decision

Use the **Strict Parity Maintenance Reset** approach for the V1 PRD. V1 preserves existing content, routes, visual style, bilingual reading experience, syntax highlighting, image behavior, SEO/social metadata, and social sharing buttons. V1 removes RSS, AdSense, newsletter confirmation pages, and broken discussion links. V1 adds dark/light mode as a contained user-facing correction: system preference is the default, and the user can switch modes at any time.

## Alternatives Considered

### Alternative 1: Minimal Migration First

- **Description**: Migrate posts and routes only, with no dark/light mode improvement and minimal polish.
- **Pros**: Fastest path to a working replacement; lowest short-term scope.
- **Cons**: Leaves an obvious UX limitation immediately after launch; preserves dark-only behavior that the user no longer wants.
- **Why rejected**: Too narrow. The dark/light mode correction is a small, valuable user-facing requirement and should be part of the maintenance reset.

### Alternative 2: Reader Experience Refresh

- **Description**: Use the migration to improve content discovery and reading experience with search, RSS restoration, updated sharing, layout refresh, and other reader-facing additions.
- **Pros**: Higher visible reader value; more complete modernization.
- **Cons**: Increases regression risk; violates the strict migration scope; delays removing Gatsby technical debt.
- **Why rejected**: V1 must prioritize getting off Gatsby with confidence. Reader growth features belong in V2 after the baseline is stable.

## Consequences

### Positive

- Keeps the PRD focused on a bounded, launchable maintenance outcome.
- Protects existing readers from broken URLs or visual regressions.
- Removes unused legacy surfaces instead of carrying them forward.
- Allows one high-confidence UX improvement that aligns with current user preference expectations.

### Negative

- Defers search, RSS restoration, newsletter, redesign, and broader personal-site expansion.
- Requires careful parity review because the success standard is strict.
- Dark/light mode adds a small amount of product surface beyond a pure migration.

### Risks

- **Parity risk**: Strict URL and visual parity create a high bar. Mitigation: require manual checklist, automated URL audit, and Lighthouse release gate.
- **Scope creep risk**: The dark/light mode improvement could invite other improvements. Mitigation: explicitly mark all non-essential reader features as out of scope for V1.

## Implementation Notes

The PRD should describe dark/light mode as a user capability and release requirement only. Technical implementation details belong in the TechSpec.
