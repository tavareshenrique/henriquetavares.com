# ADR-005: Use Static Share Links and Focused Migration Validation

## Status

Accepted

## Date

2026-05-16

## Context

The current Gatsby site includes social sharing UI and a stale `overreacted.io` discussion URL. The PRD requires preserving useful social sharing while removing broken discussion links. It also requires launch validation through manual parity review, automated URL audit, and Lighthouse, but the user selected a focused automated validation scope for the TechSpec: URL/content audit plus lint/build only.

## Decision

Replace the existing social sharing implementation with static share links and remove the broken discussion link entirely. Do not retain `react-share` or hydrate a React island for sharing. For automated validation, include lint/build plus a URL/content audit script that verifies expected route count, content count, paired translations, and absence of removed/forked surfaces. Keep Lighthouse and visual parity checks as release checklist activities rather than required automated CI tasks in V1.

## Alternatives Considered

### Alternative 1: Keep `react-share` as a Hydrated Island

- **Description**: Preserve the current sharing library and hydrate a small React component in Astro.
- **Pros**: Closer implementation continuity; likely similar UI with less custom markup.
- **Cons**: Adds a React dependency and hydration for simple outbound links; works against the minimal-JS migration goal.
- **Why rejected**: Static links provide the required user capability with less JavaScript and fewer dependencies.

### Alternative 2: Add Screenshot Regression and Lighthouse CI Automation

- **Description**: Automate visual regression screenshots and Lighthouse CI in addition to route/content checks.
- **Pros**: Stronger automated release confidence.
- **Cons**: More setup and maintenance cost; heavier than necessary for the first migration TechSpec.
- **Why rejected**: V1 should keep automated validation focused. Visual and Lighthouse validation remain required release activities but do not need full automation in the initial implementation plan.

## Consequences

### Positive

- Reduces client JavaScript and dependency surface.
- Removes the stale discussion behavior cleanly.
- Keeps validation work practical and directly tied to migration risk.
- Provides deterministic checks for route/content parity and legacy-surface removal.

### Negative

- Static share links may not match every behavior of `react-share`.
- Lighthouse and visual regression checks require manual or separate release execution unless automated later.

### Risks

- **Share parity risk**: Static links may differ slightly from existing share popups. Mitigation: validate representative share URLs manually before launch.
- **Validation gap risk**: Without automated screenshots, visual drift may slip through. Mitigation: require manual visual parity checklist for representative pages.

## Implementation Notes

The TechSpec should include a validation script that fails on missing routes/content, unexpected RSS/newsletter/ad/discussion references, and stale `overreacted.io` references in generated output or source files intended for V1.
