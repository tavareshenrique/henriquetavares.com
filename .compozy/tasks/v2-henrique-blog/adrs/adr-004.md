# ADR-004: Rebuild Styling with Tailwind and a Minimal Theme Script

## Status

Accepted

## Date

2026-05-16

## Context

The current Gatsby site uses Typography.js, SCSS, CSS variables, and a forced dark-only body class. The PRD requires the migrated site to feel like the same blog without turning the work into a redesign. Exact Typography.js runtime parity is not required; the user is open to styling changes that stay as similar as practical and make the migration easier. The user selected Tailwind as the styling system for the migration, and selected a tiny pre-paint script with `localStorage` override and system preference fallback for theme persistence.

## Decision

Use Tailwind for the migrated site's styling while preserving close visual continuity with the current design. Do not carry forward Typography.js runtime styling or recreate its rhythm/scale helpers exactly. Replace it with Tailwind theme tokens and global CSS for typography, article content, code highlighting, and practical visual similarity. Implement dark/light mode with a small inline script that runs before paint, reads a persisted user override from `localStorage`, falls back to `prefers-color-scheme`, and sets a root class/data attribute consumed by Tailwind styles.

## Alternatives Considered

### Alternative 1: Port Generated Typography.js Rules to Plain SCSS

- **Description**: Keep the current visual rules as plain SCSS/CSS tokens and avoid introducing Tailwind.
- **Pros**: Lowest visual continuity risk; fewer new styling concepts.
- **Cons**: Preserves more Gatsby-era styling assumptions; misses the user's preference for Tailwind; less ergonomic for future styling changes.
- **Why rejected**: Tailwind was selected as the preferred styling direction, and strict parity can still be achieved with global prose/code styles.

### Alternative 2: Hydrated React Theme Component

- **Description**: Use React state to manage theme selection and persistence.
- **Pros**: Familiar component state model; easy to build a rich toggle.
- **Cons**: Adds unnecessary client JavaScript and risks theme flash before hydration.
- **Why rejected**: A pre-paint inline script is smaller, faster, and better aligned with Astro's minimal-JS model.

## Consequences

### Positive

- Modernizes styling without preserving Typography.js runtime dependencies.
- Enables dark/light mode without shipping a React hydration island for theme state.
- Keeps the theme decision available before first paint, reducing flash risk.
- Gives future design work a clearer Tailwind foundation.

### Negative

- Tailwind migration increases visual parity review burden.
- Typography and article styles must be carefully mapped instead of copied directly.
- Light-mode tokens need to be created because the current site effectively ships dark-only styling.

### Risks

- **Visual drift risk**: Rebuilding styling in Tailwind can subtly alter typography and spacing. Mitigation: include representative visual continuity checks for index pages, code-heavy posts, image-heavy posts, and mobile; accept minor differences that simplify migration and do not harm readability.
- **Theme flash risk**: If the inline script runs too late, visitors may see the wrong theme briefly. Mitigation: place the script in the base document head before rendered content styles apply.

## Implementation Notes

Use Tailwind utilities for layout and components, but keep global article/prose and code-highlight styles where utility classes would make Markdown rendering brittle.
